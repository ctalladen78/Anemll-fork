## 3. Challenged & Technical Blockers

### 3.1 CoreML Serialization Hangs
The primary challenge was a persistent hang during the "Saving" phase of `ct.convert`. Even for 3-layer chunks, the CoreML 9 compiler would stall indefinitely while writing the internal Protobuf representation.
- **Root Cause**: Excessive MIL graph complexity and `StateType` metadata bloat triggering extreme memory swap (thrashing) on the M4.

### 3.2 Memory Thrashing
System diagnostics revealed `Swapouts` exceeding 160M pages (approx. 2.7TB of disk thrashing). This I/O bottleneck effectively locked the workspace during large model conversions.

### 3.3 Circular References
Initial attempts at stateful tracing failed due to `RecursionError` in PyTorch's module traversal when linking parent modules to state providers.

---

## 4. Methodologies: Tried vs. Succeeded

| Methodology | Result | Reason |
| :--- | :--- | :--- |
| **Monolithic State-Based Trace** | **FAILED** | Immediate Protobuf serialization hang. |
| **6-Layer Stateful Chunking** | **FAILED** | Memory pressure too high for M5 (32GB) during serialization. |
| **Stateful 3-Layer Chunking** | **FAILED** | Still triggered serialization hang despite smaller size. |
| **Stateless 1-Layer Fragment** | **SUCCESS** | **Guaranteed path to disk.** Each layer is ~50MB, bypassing I/O locks. |

### The Success Path: Stateless Fragments
The winning methodology involved treating each of the 24 layers as a standalone, stateless mathematical function:
1. **Trace**: `Forward(Inputs, States_In) -> (Outputs, States_Out)`
2. **Standard Tensors**: Use standard `TensorType` for states during conversion to keep the MIL graph lean.
3. **External Chaining**: The fragments are linked in a Swift/Python loop, which produces a functionally equivalent stateful model with none of the compiler overhead.
